# CUTE Error Code System (초안)

목표: 개발 중 발생하는 에러를 임의 문구가 아니라 **코드 기반**으로 관리해, 원인 분석/재현/수정/문서화를 빠르게 만든다.

---

## 1) 왜 필요한가

현재 메시지 예:
- "실시간 스트림 연결 실패: backend CORS_ORIGINS ..."

문제:
- 사용자가 원하는 메세지와 개발자가 원하는 메세지가 다름
- 사용자 예시 : 주소가 올바르지 않습니다. IP 설정을 확인하세요
- 개발자 예시 : 벡엔드 CORS 주소가 올바르지 않습니다. 프론트 IP를 벡엔드에서 허용해 주세요.
- 개발 중 발생한 버그는 원인 경로가 DB, 벡엔드 서버, 유닛 서버 등 다양할 수 있음.
- 원인 경로가 불확실하면 동일 에러 재발생 시에 조치가 느리고, 코드 외적인 원인(DB, 드라이버, 네트워크)을 찾는 건 AI도 느림.
- 개발 중 발생한 버그에 원인 경로를 적어두면 에러 조치가 신속해짐
- 런타임 버그를 추적하는 것은 아직 잘 모르겠음. 개발 중 예상하지 못하는 버그에 대해서는 대책이 구체적으로 없음
- 런타임을 장기간 확보하기 위해 서버 PC를 구동하고, 지속적으로 확인하는 방향이 필요할 것으로 보임. AI를 활용해서 실행을 chat 안에서 하고, 감시하도록 하면 좋을 것 같음.
- 배포 후 발생한 버그는 사용자가 어떤 동작을 해서 발생했는지 추적하는 것으로 디버깅 난이도를 낮출 수 있음
- 사용자 정보, 동작 정보, 시간 등의 항목을 수집하면 됨 
- UI/로그/문서가 1:1로 연결되지 않음

해결:
- 모든 에러에 **고유 코드** 부여
- UI에는 `코드 + 사람 친화 메시지`
- 로그에는 `코드 + 기술 상세`

---

## 2) 코드 규칙

형식(간소화):
- `<번호 4자리>`

예시:
- `1000` (IP 오류)
- `1001` (CORS 오류)
- `3001` (프론트 제어 요청 실패)

대역:
- `1000`대: 네트워크/접속 (IP/CORS/스트림)
- `2000`대: 유닛/데이터/스키마 (순서는 뒤에서 세부 확정)
- `3000`대: 프론트(UI/상태/제어)
- 추후 `4000`대 이상 확장 가능

규칙:
1. 같은 원인은 같은 코드 재사용
2. 코드 재사용 시 메시지는 변경 가능 (의미는 유지)
3. 삭제 대신 `deprecated: true` 처리
4. 코드 접두어(`CUTE`, `NET`)는 사용하지 않는다

## 7) 운영 원칙

- 에러 문구를 바꾸기 전에 코드 먼저 정한다.
- 코드 없는 에러는 PR에서 반려한다.
- runbook 링크 없는 error code는 TODO로 표시한다.
- DB보다 **JSON 카탈로그 단일 파일**을 source of truth로 유지한다.
- 코드 생성/수정 시 에러 코드 규칙을 프롬프트/리뷰 체크리스트에 고정한다.

- [ ] `error-catalog.json` 등록됨

- [ ] userMessage + devHint 분리됨
